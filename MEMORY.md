## 📂 Architectural Decisions
*(Log specific choices made during the build here so future agents respect them)*
- **2026-07-23 — Astro 7.x, not 5.x:** `npm create astro@latest` installed Astro ^7.1.3 (docs originally said 5.x). Confirmed with the human to proceed on latest rather than pin to 5.x. `AGENTS.md` and `agent_docs/tech_stack.md` updated to say 7.x.
- **2026-07-23 — Tailwind 4 is CSS-first, no `tailwind.config.mjs`:** `astro add tailwind` wires up `@tailwindcss/vite` in `astro.config.mjs`; design tokens (Dawn Light palette, fonts) live in an `@theme` block in `src/styles/global.css` instead. Docs updated accordingly.
- **2026-07-23 — Dark mode deferred:** Only light-mode Dawn Light tokens were added to `@theme` in Phase 1. The `[data-theme="dark"]` override values and the toggle script are Phase 3 scope per the roadmap — not added yet.
- **2026-07-23 — Header nav grows with the site:** `Header.astro` started linking only the site name back to `/`; added "Now" (Step 4), then "Writing" and "Projects" (Steps 6-7) as each page went live. Nav is now complete for all Phase 2 sections.
- **2026-07-23 — `projects` schema left without a `draft` field.** The open question from Step 3 was resolved by inaction: Step 7 built the Projects index using the schema as-is (no `content.config.ts` change, which is a Protected Area). `projects/ayosotomi-com.md` is now genuinely public — acceptable since its content is 100% factual. If future project write-ups need a staging/review period before publishing, `projects` will need the same `draft: z.boolean().default(false)` field `blog` has — ask the human before adding it (schema change).
- **2026-07-23 — Blog post routes use `[...slug].astro`, not `[slug].astro`.** Nested content IDs (e.g. `essays/on-quiet-building`, from files under `blog/essays/`) contain `/`. A single dynamic segment can't hold that — Astro's rest-parameter syntax (`[...slug]`) is required for any collection with subfolders. `projects` has no subfolders, so `projects/[slug].astro` (no rest param) is correct there. **If more nested blog subfolders get added later, this still works — but don't "clean up" the Writing route back to `[slug].astro`, it will break.**
- **2026-07-23 — `site` set in `astro.config.mjs`:** `site: 'https://ayosotomi.pages.dev'` added so `@astrojs/rss` (new dependency, Step 6) generates absolute URLs instead of relative ones. Update this if/when a custom domain replaces the `.pages.dev` one.
- **2026-07-23 — `z` now imported from `astro/zod`, not `astro:content`:** Astro 7 deprecated `import { z } from 'astro:content'` in favor of `import { z } from 'astro/zod'` (which re-exports Zod v4). Also switched `z.string().url()` → `z.url()` (Zod v4's top-level format validators). `npx astro check` confirms 0 errors/warnings/hints with this setup.
- **2026-07-23 — Sample content: drafted, then un-drafted, then made public.** 3 blog posts were written `draft: true` (Step 3, since they're AI-written in the site owner's voice), flipped to `draft: false` after human review (Interlude, before Step 4), and are now genuinely live and public since Step 6 built the Writing pages that render them. If real posts get added later, remember `draft: true` is the mechanism to stage them privately first.
- **2026-07-23 — Header/Footer need `w-full` alongside `mx-auto max-w-2xl`.** `<body>` is `flex flex-col` (Phase 1, to pin the footer down on short pages), which makes `Header`/`Footer` flex items. A flex item with an auto margin on its cross axis does **not** stretch to the container width even under the default `align-items: stretch` — the auto margin absorbs the free space instead, and the item shrinks to fit-content. This meant `header`/`footer` never actually reached 672px; they collapsed to exactly their content's width, so `justify-between` had zero slack and items touched (0px gap). Latent since Phase 1 but invisible until Step 4 added a 2nd header child (`<nav>`) for `justify-between` to actually operate on. **Fix:** add `w-full` (explicit `width: 100%`, capped by `max-width`) so `width` is no longer `auto` — this sidesteps the stretch/auto-margin conflict. **Rule going forward: any direct child of `<body>` using `mx-auto max-w-2xl` needs `w-full` too, since `<body>` is a flex column.** Also added a `flex-col`/`sm:flex-row` responsive stack to `Header.astro` — separately, the full wordmark + 3 nav links don't fit in ~327px of mobile width with any gap, which is a content-density issue, not the same bug.
- **2026-07-23 — Category-scoped prev/next post nav:** `writing/[...slug].astro` sorts posts within the *same category only* (not the whole `blog` collection) by `publishedAt`, finds the current post's index, and links to its immediate neighbors. `PostLayout.astro` hides the nav entirely when a post has no neighbor in a direction. Visually confirmed once the "essay" category had 2+ posts (after the Substack migration published) — "You Might Be Somebody's Illusion" correctly shows "← Previous / Why They Join".
- **2026-07-23 — Substack HTML export has (at least) three different "special content" div structures, not two.** Beyond `image-gallery-embed` and `captioned-image-container` (documented above), there's also `<div class="callout-block" data-callout="true"><p>...</p></div>` for pull-quote-style callouts. A converter that doesn't explicitly handle a given div class silently drops it — no error, content just vanishes. Missed this on the first pass; only "You Might Be Somebody's Illusion" used it (confirmed via grep across all 4 posts), and the human caught the gap by comparing the published post against a screenshot of the real Substack page. Fixed by patching the already-published Markdown directly (not regenerating, which would have discarded manually-added blockquote edits) and updating the conversion script for future use. **Lesson: if more Substack content ever gets migrated, grep the raw export HTML for div class names first rather than assuming the two previously-seen structures are exhaustive.**
- **2026-07-23 — Blockquote styling added to `PostLayout.astro`:** `prose-blockquote:*` Tailwind Typography modifiers (serif font, `text-xl`, not italic, accent-colored left border) make real `<blockquote>` elements read as distinct pull-quotes instead of blending into body prose. Substack's export HTML has **no `<blockquote>` tag anywhere** — pull-quote-style emphasis there is just a styled `<em>`-wrapped paragraph, so promoting the right lines to real Markdown `> ` blockquotes in the migrated posts required editorial judgment (which lines read as "pull quote worthy"), not a mechanical conversion rule.
- **2026-07-23 — Substack migration: completed via human-provided export, not scraping.** Live browser-based extraction was correctly abandoned mid-attempt (copyright-reproduction limits on autonomous scraping, regardless of claimed authorship). The human then provided `substack_export.zip` (their own Settings → Exports download) — a fundamentally different situation (human directly supplying their own document). `posts.csv` in the export has an `is_published` column, which was used to confirm scope: exactly 5 published posts exist (the 4 targets + the explicitly-excluded Nigerian-student one); 3 unpublished drafts and a theme-preview file that never appeared on the public archive were correctly left out. Substack represents post images two different ways in its export HTML (`image-gallery-embed` JSON-in-attribute divs, and `captioned-image-container` figure/anchor structures) — a converter handling only one silently produces zero images for posts using the other, with no error thrown. All 4 posts are in `src/content/blog/essays/`, `draft: true`, images co-located as sibling `.jpeg` files (Astro's relative-path image handling processes these automatically once a post is un-drafted and actually rendered — verified this by temporarily un-drafting one image-heavy post, confirming real WebP optimization output, then reverting).
- **2026-07-23 — ProjectCard fanned-deck stacking uses `isolate`, not negative `z-index`.** First attempt used `-z-10`/`-z-20` on the two background layers; caught before shipping that negative z-index on a child of a `position: relative` (but not stacking-context-establishing) parent can escape to a shared ancestor context and interact unpredictably with sibling cards in the grid. Fixed by adding `isolate` to the card wrapper (creates a proper stacking context regardless of z-index) and using small **positive** z-index values (`z-0`, `z-[1]`, `z-10`) scoped inside it instead.
- **2026-07-23 — Cloudflare: classic Pages, not Workers Builds.** First connection attempt used Cloudflare's newer Workers Builds pipeline (`npx wrangler deploy`), which auto-ran `astro add cloudflare` non-interactively inside the CI container since no `wrangler.jsonc` existed — this silently added the `@astrojs/cloudflare` adapter plus a KV namespace (`SESSION`) and Images binding (`IMAGES`) that the site doesn't use, and deployed to a `workers.dev` subdomain. None of that landed in git (repo stayed plain static). Reconnected via classic Cloudflare Pages instead (dashboard project type, not Workers Builds) — deploy log confirms "No Wrangler configuration file found," plain `dist/` static asset upload, no adapter, no bindings. Live at https://ayosotomi.pages.dev/. **If reconnecting Cloudflare again, use the Pages project type, not Workers Builds/Wrangler**, to avoid this auto-adapter behavior recurring.
